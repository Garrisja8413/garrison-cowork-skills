# TRIAL-BRIEF Output Format — BODY-ONLY Contract

## Purpose

This reference defines the exact output format for the TRIAL-BRIEF skill.
All modes produce output conforming to this contract. The output is
BODY-ONLY text ready for NML formatting overlay and SA shell integration.

## BODY-ONLY Principle

The TRIAL-BRIEF skill produces the substantive body of the trial brief
only. The following components are NOT included in the output:

| Excluded Component | Source |
|-------------------|--------|
| Caption / case header | SA shell (merge tokens) |
| Court name and case number | SA shell |
| Document title ("PLAINTIFF'S TRIAL BRIEF") | SA shell |
| Table of contents | Generated by NML from headings |
| Table of authorities | Generated by NML from citations |
| Signature block | SA shell |
| Certificate of service | SA shell |
| Verification / affidavit | SA shell (if required) |
| Page numbers | SA shell / NML formatting |

The output begins with Section I (Introduction) and ends with Section VIII
(Conclusion). Everything between is BODY-ONLY prose.

## Output Contract — Required Sections (in order)

The substantive brief text, organized as follows:

```markdown

[Introduction text — 1 page maximum]
[No legal citations; narrative summary of case and strongest evidence]
[Use client name or SA merge token, not "plaintiff"]

## II. Statement of the Case

[Procedural history with docket references]

### B. Nature of Claims

[Claims remaining for trial]

### C. Parties

[Party identification]

## III. Statement of Facts

### A. Background

[Pre-incident facts — humanize the client]
[Every fact cited: (Ex. [#], Bates [####]) or ([Name] Depo. [Page]:[Line])]

### B. The [Incident]

[Chronological incident narrative with record citations]

### C. Injuries and Medical Treatment

[Medical narrative with record citations]

### D. Impact and Damages

[Damages impact narrative with record citations]

## IV. Legal Standards

[Preponderance standard — UJI 13-201]

[Elements with UJI and case authority]

### C. [Cause of Action 2] (if applicable)

### D. Damages Standards

## V. Argument

#### 1. [Element 1]

### B. [Cause of Action 2] (if applicable)

### B. Lost Wages and Earning Capacity

[Summary and relief requested]
[No signature block — SA shell provides]
```

### Part 2: Divider

```markdown
---
```

A single horizontal rule separates the brief body from the analytical
sections. The divider signals to downstream systems that the BODY-ONLY
prose has ended.

### Part 3: Cite Table

Every factual and legal assertion in the brief must appear in the Cite
Table with its source.

```markdown
## Cite Table

| # | Assertion | Evidence (CFP Fact#, LPB LawTag + Pinpoint, PCM Element_ID) | Type | Confidence | Notes |
|---|-----------|-------------------------------------------------------------|------|------------|-------|
| 1 | Defendant ran the red light | CFP Fact#7 (Ex. 3, Police Report, Bates 0001-0005) | Factual | HIGH | Undisputed |
| 2 | Duty of ordinary care | LPB-L023 (UJI 13-1601 NMRA) | Legal | HIGH | Standard instruction |
| 3 | Medical expenses: $45,000 | CFP Fact#22 (Ex. 10-15, Medical Bills); DC-D001 | Factual | HIGH | Verified amounts |
| 4 | Causation opinion | CFP Fact#18 (Ex. 8, Dr. Jones Report, Bates 0340) | Expert | MEDIUM | [VERIFY] depo designation |
```

Column specifications:

| Column | Content | Rules |
|--------|---------|-------|
| # | Sequential number | Continuous numbering through the table |
| Assertion | The factual or legal claim made in the brief | Brief but specific |
| Evidence | Pack reference with pinpoint | CFP Fact#, LPB LawTag, PCM Element_ID, DC reference |
| Type | Assertion type | `Factual`, `Legal`, `Expert`, `Procedural`, `Stipulated` |
| Confidence | Verification confidence | `HIGH` (verified), `MEDIUM` (partially verified), `LOW` (unverified) |
| Notes | Flags and status | `[VERIFY]`, `[VERIFY-CITE]`, `[ELEMENT-RISK]`, etc. |

### Part 4: Gate Results

```markdown
## Gate Results

- **Traceability**: [PASS/FAIL] — All facts traced to CFP Fact#; all law traced to LPB LawTag
  - [If FAIL: List assertions without pack source]
- **Element Coverage**: [PASS/FAIL] — Every element of every COA addressed in Argument section
  - [If FAIL: List missing elements with Element_ID]
- **Status Gate**: [PASS/FAIL] — All CFP rows used are DRAFT-READY or VERIFIED; all LPB rows are PackReady=Yes
  - [If FAIL: List non-ready rows used]
- **Shepard Gate**: [PASS/FAIL] — All LPB case citations Shepardized with no negative treatment
  - [If FAIL: List cases with negative treatment or unverified Shepard status]
- **Record Discipline**: [PASS/FAIL] — Every material fact in Statement of Facts has exhibit or deposition cite
  - [If FAIL: List facts without record citation]
- **NML Compliance**: [PASS/FAIL] — Output conforms to NML formatting standards
  - [If FAIL: List formatting issues]
- **Conflicts**: [Note any conflicts between pack sources or inconsistent facts]
- **Merge-token integrity**: [PASS/FAIL] — All SA merge tokens preserved exactly
  - [If FAIL: List modified or missing merge tokens]
```

Gate evaluation rules:
- Traceability: FAIL if ANY assertion lacks a pack source reference
- Element Coverage: FAIL if ANY element of any active COA is not addressed
- Status Gate: FAIL if ANY non-ready pack row is used for a material assertion
- Shepard Gate: FAIL if ANY case citation has unresolved negative treatment
- Record Discipline: FAIL if ANY material fact in Section III lacks a record cite
- NML Compliance: FAIL if output uses formatting incompatible with NML
- Merge-token integrity: FAIL if ANY merge token is modified or missing

### Part 5: Next Requests

```markdown
## Next Requests

1. [Specific request — e.g., "Provide exhibit list with Bates ranges for Section VII"]
2. [Specific request — e.g., "Confirm deposition designations for [witness name]"]
3. [Specific request — e.g., "Provide updated medical billing summary for damages section"]
4. [Specific request — e.g., "Confirm trial date and filing deadline for trial brief"]
```

Next Requests identify what the attorney or paralegal needs to provide
to complete or improve the brief. Each request should be specific and
actionable.

---

### Part 6: Open Items

```markdown
## Open Items

### [VERIFY] Items
- [ ] Ex. 8 (Dr. Jones Report) — confirm this exhibit is on the exhibit list
- [ ] Deposition of [defendant] — confirm pages designated for trial

### [VERIFY-CITE] Items
- [ ] *Smith v. Jones*, 2020-NMCA-xxx — Shepardize and confirm current
- [ ] UJI 13-1604 text — verify against current UJI edition

### [ELEMENT-RISK] Items
- [ ] Negligence — Causation (E3): Expert opinion needed; currently MARGINAL support
- [ ] Damages — Future lost wages (D4): No economist designated

### [DECISION REQUIRED] Items
- [ ] Whether to include punitive damages argument (requires attorney approval)
- [ ] Whether to address comparative fault preemptively or wait for defense brief
- [ ] Specific damages amounts to include in brief vs. reserving for trial
- [ ] Whether to attach proposed jury instructions as exhibit to trial brief
```

Each open item must be:
- Actionable (someone can resolve it)
- Specific (identifies exactly what needs attention)
- Categorized by type (VERIFY, VERIFY-CITE, ELEMENT-RISK, DECISION REQUIRED)
- Presented as a checkbox for tracking

## Mode-Specific Output Variations

### FULL BRIEF Mode (Default)

Produces all sections (I-VIII) plus all analytical sections. Complete output.

### LEGAL MEMO Mode

Output differences:
- Omit Section III (Statement of Facts) — or abbreviate to relevant facts only
- Omit Section VI (Damages) — unless the legal issue concerns damages
- Emphasize Section IV (Legal Standards) and Section V (Argument)
- Title the document as "MEMORANDUM OF LAW" rather than "TRIAL BRIEF"
- Narrower scope — address only the specific legal issues identified

### SUPPLEMENT Mode

Output differences:
- Begin with a brief statement identifying the original brief and the
  reason for supplementation
- Include only new or updated sections
- Reference the original brief by date and docket number
- Include an updated Cite Table covering only new citations

### SECTION DRAFT Mode

Output differences:
- Produce only the requested section(s)
- Include a Cite Table limited to the drafted section
- Include Gate Results limited to the drafted section
- Note dependencies on other sections as `[CROSS-REFERENCE: Section [X]]`

## Formatting Standards

1. **Markdown**: Use markdown throughout. No HTML.
2. **Headings**: `##` for main sections (I, II, III...), `###` for
   subsections (A, B, C...), `####` for sub-subsections (1, 2, 3...).
3. **Citations in text**: Parenthetical format — `(Ex. [#], Bates [####])`
   or `([Name] Depo. [Page]:[Line])` or `*[Case Name]*, [citation]`.
4. **Block quotes**: Use markdown `>` for quotations from cases, statutes,
   or instructions.
5. **Emphasis**: Bold for section headings and key terms. Italic for case
   names. No underlining.
6. **Tables**: Markdown table syntax with headers.
7. **Lists**: Numbered for sequential items, bulleted for non-sequential.
8. **Flags**: Bracketed all caps — `[VERIFY]`, `[VERIFY-CITE]`,
   `[ELEMENT-RISK]`, `[DECISION REQUIRED]`.

## Token Fail-Safe Chunking

If the complete output exceeds token limits, chunk as follows:

| Chunk | Contents |
|-------|----------|
| Part 1 | Sections I-III (Introduction, Statement of Case, Statement of Facts) |
| Part 2 | Sections IV-V (Legal Standards, Argument) |
| Part 3 | Sections VI-VIII (Damages, Evidentiary Issues, Conclusion) |
| Part 4 | Cite Table + Gate Results + Next Requests + Open Items |

Each chunk must begin with a header identifying which part it is and
how many total parts exist:

```markdown
## TRIAL BRIEF — Part [X] of [Y]
```
